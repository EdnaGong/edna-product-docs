# 深度选车功能设计讨论

> 创建时间：2026-04-29
> 状态：讨论中

---

## 一、核心问题定义

### 主对话 vs 深度选车的本质差异

| 维度 | 主对话（看选买用） | 深度选车 |
|------|-------------------|----------|
| 目标 | 解决用户单次 query | 帮用户完成"选到合适的车"这件事 |
| 生命周期 | 单轮或短多轮 | 跨轮次、跨时间段的长期任务 |
| 驱动力 | 用户问，Agent 答 | Agent 主动推进，用户确认 |
| 上下文 | 对话上下文 | 用户画像 + 需求状态机（持久化） |
| 结束信号 | 用户停止提问 | 用户明确"就选这辆" |
| 核心价值 | 信息获取效率 | 决策质量 + 决策信心 |

---

## 二、选车的用户旅程拆解

选车不是一次问答，是一个**决策收敛过程**，分以下阶段：

```
阶段1: 需求模糊  →  阶段2: 需求明确  →  阶段3: 备选缩圈  →  阶段4: 深度对比  →  阶段5: 决策确认
```

### 各阶段用户状态

**阶段1 需求模糊**
- 用户状态：不知道自己想要什么，或只有大方向（"家用，预算20万"）
- 核心任务：挖掘隐性需求，帮用户建立需求框架
- Agent 行为：结构化提问 → 工况推理 → 生成用户画像

**阶段2 需求明确**
- 用户状态：知道自己要什么，但不知道哪款车最合适
- 核心任务：将需求映射到车型，给出有依据的推荐
- Agent 行为：需求 → 条件 → 推荐 N 款（含推荐理由）

**阶段3 备选缩圈**
- 用户状态：手里有几个备选，需要淘汰
- 核心任务：基于用户需求排除不匹配车型
- Agent 行为：对比关键维度差异，给出淘汰建议（而不是罗列参数）

**阶段4 深度对比**
- 用户状态：锁定 2~3 款，需要做最终决策
- 核心任务：在用户最在意的维度上深度分析
- Agent 行为：聚焦用户 TOP 关注点做深度分析，而不是全维度比较

**阶段5 决策确认**
- 用户状态：基本决定了，需要信心确认
- 核心任务：打消顾虑，强化决策信心
- Agent 行为：总结选择理由 + 主动提示下一步（试驾/优惠/经销商）

## 三、设计方向探讨

### 方向A：任务型 Agent（推荐重点讨论）

**核心理念**：把"选车"当作一个有状态的长期任务，Agent 是任务执行者，用户是决策者。

**关键设计点**：

1. **需求状态机（持久化）**
   - 用户画像字段：预算 / 城市 / 用途 / 家庭情况 / 痛点偏好 / 用车工况
   - 每轮对话后更新状态，下次打开直接继承
   - 状态可视化给用户看（让用户感知"Agent 记住我了"）

2. **Agent 主动推进**
   - 不等用户问，主动识别当前阶段
   - 每轮结束主动给出"下一步建议"（"你要不要告诉我你主要在城市还是高速开？"）
   - 避免无目的的多轮，每轮都有推进感

3. **结构化提问策略**
   - 不能连续追问超过 3 个问题
   - 每次只问最关键的 1~2 个未知变量
   - 优先问区分度高的（用途 > 预算 > 品牌偏好）

4. **推荐机制**
   - 推荐要有理由，理由要锚定用户说过的话（"你说主要高速跑，所以我更推荐 X 的 NVH 表现"）
   - 不要求全，要求准
   - 推荐 3~5 款，分主推+备选，而不是全量列表

5. **动态收敛**
   - 随着对话深入，推荐范围越来越窄
   - 明确呈现"从 20 款 → 8 款 → 3 款"的收敛过程，用户有掌控感

---

### 方向B：选车向导（线性 Checklist）

**核心理念**：把选车拆成固定步骤，用户一步步填表，最后出推荐。

**优点**：结构清晰，开发简单，容易标准化

**缺点**：体验僵硬，像问卷不像对话，用户流失风险高；无法处理用户中途改变想法的情况

**结论**：可作为冷启动 MVP，但长期不是理想形态

---

### 方向C：选车助手（弱引导，强工具）

**核心理念**：用户随时来随时问，Agent 提供工具（对比/筛选/排行），不主动推进任务

**优点**：自由度高，不强迫用户

**缺点**：回到了"问什么答什么"的逻辑，没有解决"选车是长期任务"的核心问题

**结论**：这是主对话已经做的事，深度选车不能重复

- 存在哪里？对话历史 or 独立的用户需求档案？
- 用户能不能手动编辑自己的需求档案？
- 多次进入深度选车，是续上还是重开？

- Agent 每轮都主动追问，用户会不会觉得烦？

- 用户一直不回答关键问题怎么办？

### Q4：推荐结果的形式

- 纯文字推荐 vs 卡片式展示（带车图、关键参数、匹配理由）

- 取舍标准是什么？

## 五、初步判断与建议

**核心判断**：深度选车的本质不是"更好的问答"，是**一个有状态、有目标、会主动推进的任务 Agent**。

**设计重心应该放在**：
1. **状态持久化** — 用户画像和需求档案跨轮次保存，是一切的基础
2. **阶段感知** — Agent 能判断用户在哪个阶段，用不同策略响应
3. **主动性** — 不等用户问，主动给出下一步

**MVP 建议路径**：

```
MVP1: 需求收集 + 初步推荐（阶段1→阶段2）
  - 结构化提问（6~8 个关键变量）
  - 基于需求给出 3~5 个推荐 + 推荐理由
  - 用户画像持久化（本地 or 账号绑定）

MVP2: 备选对比（阶段3→阶段4）
  - 用户圈定 2~3 款，基于用户关注点做对比
  - 对比结果可保存

MVP3: 全链路打通
  - 完整五阶段 + 和主对话的互通
```

## 六、已确认的设计决策

### Q1：产品形态与入口 ✅

**决策**：独立页面 + 固定入口

深度选车是独立页面，有自己的数据层和视觉空间，不是主对话的模式切换。

### Q2：用户画像持久化与可见性 ✅

**决策**：持久化 + 展示给用户 + 可编辑

**核心原则**：
- 透明化解决隐私顾虑，隐藏数据反而增加不信任感
- 展示形式用自然语言摘要，不用字段表格
- 用户可修改或重置（给控制感）
- 位置不抢眼，可折叠，主体还是对话

**示例展示形式**：
> 根据我们的对话，你的核心需求是：20万以内、家用为主、周末偶尔自驾游、最在意空间和安全性，品牌没有强偏好。

### Q3：推进节奏策略 ✅

**核心理念**：Agent 的主动性用在"澄清需求"上，不是"催促决策"上。需求清晰了，决策自然快。

#### 用户决策模型：9-4-2-1 + 非线性跳转

用户选车不是线性收敛，而是在扩张（9款）→ 粗选（4款）→ 精选（2款）→ 决策（1款）之间反复横跳。Agent 需要：
1. 识别用户当前在哪个阶段
2. 支持双向跳转（收敛 AND 扩张）
3. 不强制推进，但主动帮助突破卡点

#### 用户卡住的三种情况 + Agent 介入策略

**情况1：需求模糊，不知道自己要什么**
- 表现：用户说"20 万家用 SUV"，Agent 给了 9 款，用户说"都差不多啊"
- 根本问题：用户没有区分标准
- Agent 介入：
  - 主动给区分维度："这 9 款其实分三类：省油但动力弱的（A/B/C）、动力强但油耗高的（D/E/F）、均衡但价格贵的（G/H/I），你更倾向哪类？"
  - 用户选了一类后，Agent 立刻收敛到 3 款，同时更新需求档案（"你更在意动力"）

**情况2：有多个需求，不知道哪个更重要**
- 表现：用户说"我要空间大、油耗低、动力强、价格便宜"
- 根本问题：需求冲突，用户没有优先级
- Agent 介入：
  - 用具体场景帮用户排优先级："如果周末经常带全家出去玩，空间更重要；如果每天市区通勤为主，油耗更重要。你的用车场景更接近哪个？"
  - 用户回答后，Agent 基于优先级重新推荐，自然收敛

**情况3：在 2~3 款之间反复纠结**
- 表现：用户在 A 和 B 之间来回问，问了 5 轮还没决定
- 根本问题：用户在等一个"确定性"，但对比越细越纠结
- Agent 介入：
  - 主动给决策框架："你纠结了这么久，说明这两款在你的需求上都能打 80 分。这种情况下，我建议你去试驾一下，实际感受更重要。要不要帮你预约？"
  - 或给明确建议："A 和 B 的差异主要在动力，但你说你主要市区开，这个差异你实际用车中感知不强。我更推荐 B，因为它油耗更低，长期用车成本省 2 万。"

#### 推进策略矩阵

| 用户需求清晰度 | Agent 策略 | 主动程度 | 推荐数量 |
|--------------|-----------|---------|---------|
| 需求明确（"我要 7 座 SUV，预算 30 万，主要高速"） | 直接推荐，等用户反馈 | 低主动 | 4~5 款 |
| 需求模糊（"20 万家用车"） | 主动给区分维度，帮用户建立标准 | 高主动 | 8~10 款分档 |
| 需求冲突（"又要省油又要动力强"） | 主动用场景帮用户排优先级 | 高主动 | 先不推荐，先澄清 |
| 决策卡住（在 2 款间反复） | 主动给决策建议或引导试驾 | 中主动 | 2 款深度对比 |

#### 阶段识别与对应策略

**扩张阶段（目标：9 款左右）**
- 用户状态：需求还不够明确，或对推荐不满意，想看更多
- Agent 行为：
  - 给 8~10 款，分 3 档（强推 3 款 + 备选 4 款 + 可考虑 2 款）
  - 每款只给核心标签（"20 万 SUV，空间大，油耗低"），不展开
  - 主动问"这里面有你感兴趣的吗？还是我换个方向再找找？"
- 触发信号：用户说"还有别的吗" / "都不太喜欢" / "我想再看看"

**粗选阶段（目标：4 款左右）**
- 用户状态：从 9 款里圈出了几个，或主动说"我在 A/B/C/D 里纠结"
- Agent 行为：
  - 基于用户圈定的 4 款，做差异化总结（不是全维度对比）
  - "这 4 款的主要区别是：A 空间最大但油耗高，B 最省油但动力弱，C 均衡但价格贵，D 性价比高但品牌力弱"
  - 主动问"你最在意哪个？"（只问 1 个维度）
- 触发信号：用户圈定 3~5 款 / 用户说"我在 X/Y/Z 里选"

**精选阶段（目标：2 款）**
- 用户状态：锁定 2 款，需要深度对比做最终决策
- Agent 行为：
  - 基于用户最在意的维度，做深度分析（不是罗列参数）
  - 比如用户在意空间，就对比"后排腿部空间 / 后备箱容积 / 实际乘坐体验口碑"
  - 给出明确倾向建议："基于你最在意空间，我更推荐 X"
- 触发信号：用户问"X 和 Y 哪个好" / 用户说"就在这两个里选了"

**决策阶段（目标：1 款）**
- 用户状态：基本确定了，需要信心确认
- Agent 行为：
  - 总结选择理由（锚定用户说过的需求）
  - 主动提示下一步："要不要帮你查查这款车的优惠 / 预约试驾？"
  - 如果用户犹豫，主动问"还有什么顾虑吗？"
- 触发信号：用户说"就这个了" / "我觉得 X 不错"

### Q4：推荐结果呈现形式 ✅

**决策**：分阶段卡片展示 + 结果保存

**不同阶段的展示形式：**
- **9 款阶段**：简化卡片（车图 + 核心标签 + 一句话推荐理由）
- **4 款阶段**：标准卡片（增加关键参数对比）
- **2 款阶段**：详细对比表（聚焦用户最在意的维度深度对比）

**结果保存**：推荐结果必须保存，否则失去"长期任务"的意义

### 任务生命周期管理 ✅

**决策**：MVP 阶段单任务模式 + 简化状态管理

#### 核心原则

1. **任务由用户控制，系统不主动中断**
   - 用户选车周期通常 1-3 个月（行业数据）
   - 长期不活跃不代表放弃，可能在观望、等发工资、等新款
   - 只有用户明确表达，任务才算完成

2. **系统只提示，不强制**
   - 用户长期未活动后回来，系统提示确认，但不自动清空数据
   - 任务数据长期保留，除非用户主动删除

#### 任务状态

```
活跃中 ←→ 已完成
```

**状态说明：**
- **活跃中**：用户正在选车，无论多久没活动都保持此状态
- **已完成**：用户明确选定了某款车

#### 用户再次进入时的处理

| 情况 | 系统行为 |
|------|---------|
| 任务活跃中，30 天内有活动 | 直接继续，不打断 |
| 任务活跃中，30 天以上未活动 | 提示"距离上次选车已经过了 X 天，要继续之前的选车，还是重新开始？"<br>选项：继续 / 重新开始 |
| 任务已完成 | 提示"上次你选的是 X，要重新选车吗？"<br>选项：重新选 / 查看上次结果 |

**30 天以上未活动时的确认策略：**
1. 先确认需求是否变化："你之前的预算是 20 万，现在还是这个预算吗？"
2. 基于确认结果：
   - 需求没变：展示之前的推荐，但提示"这些推荐是 X 天前的，我重新帮你看看有没有更新"
   - 需求变了：更新需求档案，重新推荐

#### 任务完成的唯一条件

**只有用户明确表达：**
- 用户说"就选这个了" / "我决定买 X 了"
- Agent 主动确认："确定选 Model Y 了吗？"，用户确认

**不能作为完成信号：**
- 预约试驾（试驾后可能不买）
- 查询优惠（只是了解信息）
- 长期不活跃（可能在观望）

#### 需求变更处理

**小幅调整（直接更新）：**
- 用户核心需求没变，只是细化或微调
- Agent 主动确认后更新需求档案，保留推荐历史

**重大变更（提示用户）：**
- 用户核心需求发生根本改变（之前推荐的车大部分不再适用）
- Agent 提示："你的需求和之前差别比较大，是想重新选车，还是在原来基础上调整？"
- 用户选择：重新开始 / 继续调整

**判断依据：**
- 如果新需求下，之前推荐的车大部分都不适用了 → 重大变更
- 如果新需求下，之前推荐的车还有一半以上适用 → 小幅调整
- 不设置硬编码的数字阈值（如 ±30%），由 Agent 基于上下文判断

#### 单任务 vs 多任务

**MVP 决策**：单任务模式

**原因：**
- 80% 用户一次只选一辆车
- 产品复杂度低，开发快
- 移动端体验更好

**处理方式**：用户开启新任务时，提示"检测到未完成的选车任务，是否结束？"，选项：结束旧任务 / 暂停旧任务 / 取消

**后续迭代判断标准**：
- 用户暂停任务后开新任务的比例 > 15%
- 用户反馈中提到"想同时选多辆车"的频率

### Q5：和主对话的边界 ✅

**核心判断：深度选车和主对话的区别不是"内容边界"，而是"交互模式"**

#### 内容范围

**深度选车模式支持的内容：**
- 看车：参数、配置、口碑、评测
- 选车：推荐、筛选、对比
- 买车：价格、优惠、经销商
- 用车：保养成本、保险、油耗

**只要用户还在"选车"这个任务中，所有帮助决策的信息都应该提供。**

**关键洞察：**
- 用户问价格、保养成本，不是"跑题"，而是决策依据
- 看车、买车、用车信息都是选车决策的输入
- 不应该用"内容类型"来划分边界

#### 深度选车 vs 主对话的真正差异

| 维度 | 主对话 | 深度选车 |
|------|--------|---------|
| 任务目标 | 解决单次 query | 完成"选到车"这个任务 |
| 内容范围 | 看选买用都支持 | 看选买用都支持（只要是为了选车决策） |
| 核心差异 | 问什么答什么 | 有状态、有目标、主动推进 |
| 上下文 | 对话历史 | 任务状态 + 需求档案 + 对话历史 |
| 结束条件 | 用户停止提问 | 任务完成（用户选定车型） |

**同样的问题，不同模式下的处理差异：**

**问题："Model Y 的保养费用多少？"**

- **主对话**：直接回答保养费用，结束
- **深度选车**：回答保养费用，然后主动推进："Model Y 的年均保养费用大概 3000 元，在同级别里算中等水平。理想 L7 因为是增程式，保养会便宜一些，大概 2000 元/年。这个差异对你的决策有影响吗？"

**差异：深度选车会把信息和决策任务关联起来，主动推进。**

#### 两个模式的上下文共享

**共享内容：**
- 深度选车 → 主对话：用户选定的车型、需求档案
- 主对话 → 深度选车：用户最近问过的车型、表达过的偏好

**技术实现：**

在用户上下文中维护 `cross_mode_context`：

```json
{
  "selected_car": {
    "car_id": "model_y_2024",
    "car_name": "特斯拉 Model Y",
    "selected_at": "2026-04-29",
    "source": "deep_selection"
  },
  "user_preferences": {
    "budget_range": "20-30万",
    "priorities": ["空间", "安全性"],
    "updated_at": "2026-04-29"
  },
  "recent_interested_cars": ["理想L7", "问界M7", "Model Y"]
}
```

**效果：**
- 用户在深度选车里选定 Model Y 后，回到主对话问"这车油耗怎么样"，主对话知道"这车"指 Model Y
- 用户在主对话里问过几款车，进入深度选车时，Agent 可以说"看起来你在关注 A/B/C 这几款，我帮你系统对比一下？"

#### 主对话到深度选车的引导

**原则：识别选车意图，引导但不强制**

**场景A：用户明确表达选车需求**
- 用户问："20 万家用 SUV 有哪些推荐？"
- 主对话 Agent：
  - 先给简单回答（3~5 款推荐）
  - 主动引导："如果你想深度选车，我可以帮你一步步理清需求，找到最适合你的车。要不要试试深度选车模式？"
  - 用户点击后跳转，并带上刚才的需求作为初始输入

**场景B：用户在主对话里反复问选车问题**
- 用户连续问了 3 轮选车相关问题
- 主对话 Agent：主动提示"看起来你在认真选车，要不要试试深度选车模式？我可以帮你系统地对比和筛选。"

### Q6：MVP 范围与优先级 ✅

**核心原则：先设计完整方案，再基于资源和时间做优先级排序，不因时间压力妥协体验完整性**

#### 完整方案（理想状态）

深度选车的完整能力包括：
- 独立页面 + 固定入口
- 需求档案持久化 + 可视化 + 可编辑
- 9-4-2-1 收敛模型 + 支持阶段跳转
- 基于需求清晰度的自适应推进策略
- 需求变更处理（小幅调整 / 重大变更）
- 任务状态管理（活跃中 / 已完成）
- 长期未活动的确认机制
- 和主对话的上下文共享
- 完整的看选买用信息支持

#### 分期实现计划

### P0：6月15日首次发布（核心能力）

**判断标准：没有这个能力，深度选车就不成立**

**必须具备的能力：**

1. **需求收集与档案**
   - 结构化提问（2~3 轮快速建立基础画像）
   - 需求档案生成（自然语言摘要）
   - 需求档案展示（可折叠面板，主体仍是对话）

2. **推荐与收敛（完整 9-4-2-1）**
   - 扩张阶段：推荐 8~10 款，分 3 档展示（强推 3 款 + 备选 4 款 + 可考虑 2 款）
   - 粗选阶段：基于用户反馈收敛到 4 款，给出差异化总结
   - 精选阶段：2~3 款深度对比，聚焦用户最在意的维度
   - 决策阶段：用户选定，任务完成

3. **阶段识别与跳转**
   - Agent 识别用户当前在哪个阶段（基于用户行为，不是轮次）
   - 支持扩张 ↔ 收敛的双向跳转
   - 用户说"都不满意"能回到扩张阶段，用户说"我在 X/Y 里选"能进入精选阶段

4. **主动推进（核心差异化）**
   - 识别需求模糊 → 主动给区分维度，帮用户建立标准
   - 识别需求冲突 → 主动用场景帮用户排优先级
   - 识别决策纠结 → 主动给决策建议或引导试驾

5. **任务持久化**
   - 单任务模式
   - 任务数据跨轮次保存
   - 用户下次进来能继续
   - 30 天以上未活动时，进入前确认需求是否变化

6. **基础内容支持**
   - 单车查询（参数、配置、口碑）
   - 多车对比
   - 价格、优惠信息（作为决策依据）
   - 保养成本、油耗（作为决策依据）

**为什么这些是 P0：**
- 没有完整收敛流程，就只是"更复杂的推荐"，不是"任务型 Agent"
- 没有阶段跳转，用户会觉得被强制推进，体验僵硬
- 没有主动推进，和主对话没有本质区别
- 没有任务持久化，失去"长期任务"的价值

### P1：6月15日后迭代（体验优化）

**P1 能力清单：**

1. **需求档案可编辑**
   - 用户可以直接修改需求档案（不只是通过对话）
   - 修改后 Agent 基于新需求重新推荐

2. **需求变更智能处理**
   - 区分小幅调整 / 重大变更
   - 保留需求变更历史（版本管理）
   - 用户可以回看"我之前的需求是什么"

3. **推荐结果时效性更新**

   - 用户长期未活动后回来，主动提示"要不要看看最新的"

4. **和主对话的深度集成**
   - 主对话识别选车意图，主动引导进入深度选车
   - 深度选车结果带回主对话（用户选定的车型）
   - 跨模式上下文共享（用户在主对话问过的车，深度选车能知道）

5. **用车成本详细计算**
   - 保养、保险、油耗的详细对比
   - 5 年用车成本计算器
   - 不同用车场景下的成本差异

**为什么这些是 P1：**
- 没有这些功能，核心流程仍然成立
- 这些是体验优化，不是核心差异化
- 可以基于用户反馈再优化

## 八、方案总结

### 核心定位

深度选车是一个**有状态、有目标、会主动推进的任务型 Agent**，帮助用户完成"选到合适的车"这个长期任务。

### 和主对话的本质差异

不是"内容边界"的差异，而是"交互模式"的差异：
- 主对话：问什么答什么，无状态
- 深度选车：有任务目标，有需求档案，主动推进收敛

### 核心设计原则

1. **用户决策模型：9-4-2-1 + 非线性跳转**
   - 用户选车不是线性收敛，会在阶段间反复横跳
   - Agent 支持双向跳转，不强制推进

2. **主动性用在澄清需求，不是催促决策**
   - 识别用户卡住的情况（需求模糊 / 需求冲突 / 决策纠结）
   - 主动介入帮助突破，而不是等用户自己想明白

3. **任务由用户控制，系统不主动中断**
   - 用户选车周期 1-3 个月，长期不活跃不代表放弃
   - 系统只提示确认，不自动清空数据

4. **透明化建立信任**
   - 需求档案展示给用户，用自然语言摘要
   - 推荐理由明确，锚定用户说过的需求
   - 用户有控制感（可编辑档案、可重新开始）

### 关键成功要素

1. **推荐能力**：9-4-2-1 的推荐逻辑 + 推荐理由生成
2. **阶段识别**：准确识别用户在哪个阶段，用对应策略
3. **主动推进**：识别卡点，主动介入，而不是被动等待
4. **任务持久化**：跨轮次保存，用户随时回来能继续

## 十、Agent 架构讨论（2026-04-30）

### 已确认的架构信息

- APP 整体架构：**主 Agent + 看/选/买/用四个子 Agent**
- 主 Agent 是路由和枢纽，四个子 Agent 各自负责一个阶段
- 深度选车属于选车子 Agent 范畴

### 一站式服务的核心价值

不是"功能全"，而是**上下文贯通**：
- 用户不需要重复介绍自己
- 从看车到选车到买车到用车，Agent 全程记得用户的需求、选择和原因
- 每个阶段的 Agent 都能感知用户的全局状态

### 统一用户上下文（全局共享）

```json
{
  "user_profile": {
    "基础信息": "预算20万，家用，最在意空间",
    "选车任务": { "状态": "进行中", "意向车": ["RAV4", "CR-V"] },
    "已购车": null,
    "用车记录": []
  }
}
```

### 动态 Prompt 组装机制

每次调用 API，system prompt 根据用户状态动态组装：

```
[基础人设] 固定
[用户当前状态] 动态注入：所处阶段、已选车型、需求档案
[当前阶段策略] 动态注入：当前阶段的核心任务和行为准则
[可用工具] 动态注入：当前阶段可调用的工具列表
```

### 待确认的关键架构问题

1. **子 Agent 上下文传递方式**：主 Agent 统一维护用户状态并在调用子 Agent 时注入？还是各子 Agent 自己维护状态，主 Agent 只做路由？
2. **跨子 Agent 的信息共享**：用户在选车子 Agent 里说的话（如"最在意空间"），用车子 Agent 能否感知？
3. **深度选车的定位**：是选车子 Agent 内部的一个任务型模式，还是独立的子 Agent？
